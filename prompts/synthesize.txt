You are producing the COMPRESSED context view of a coding session. You are
given a detailed "knowledge draft" (Tier-2 source facts). Your output goes
directly into the agent's active context window, so density matters.

Produce three artifacts, wrapped in tags, in this exact order:

  <facts>{JSON}</facts>
  <summary>...compressed narrative for the A view...</summary>
  <q>...refined user intent / outstanding request for the Q view...</q>

The JSON facts object must contain:
{
  "original_request": "...",
  "files_modified": [...],
  "decisions": [...],
  "current_task": "...",
  "open_issues": [...],
  "code_written": [{"name": "...", "path": "...", "purpose": "..."}]
}

Rules:
  * No information that is not already in the knowledge draft.
  * `original_request`: copy VERBATIM from the previous facts JSON if present.
    If this is the first round, extract the user's very first message verbatim.
    Truncate to 400 chars with "[…]" if longer. NEVER rephrase or summarise it.
  * The <summary> is one or two tight paragraphs — what was done, where it
    stands, what matters next. Name files, functions, and decisions
    explicitly; skip conversational texture.
  * The <q> is ONE paragraph capturing the user's still-active goals.
    * If the user provides short, conversational confirmations (e.g., 'yes',
      'ok', 'tak', 'proceed', 'continue'), DO NOT include the raw word in <q>.
      Instead, interpret the intent and describe it (e.g., 'user confirmed the
      implementation of X' or 'user agreed to proceed with Y').
    * If the knowledge draft contains a "[PREVIOUS Q_VIEW]" section, preserve it
      verbatim unless the new transcript contains an explicit new instruction
      that clearly supersedes it. Only rephrase when the previous q_view
      conflicts with `original_request` or new user instructions.
  * Under context pressure, drop execution details before dropping intent.
    `original_request` and `<q>` must survive even the most aggressive
    compaction.
  * If the draft notes that details have been elided, mention in <summary>
    that `recall_facts(query=...)` is available to expand specifics.
  * Never invent file paths, error messages, or decisions.
  * Omit conversational noise, apologies, greetings, and resolved dead-ends.
